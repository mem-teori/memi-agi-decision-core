# M.E.M.I. v12.0 — Contextual Admissibility
## Architecture Note · April 2026

---

## Bærende sætning

    The system does not learn a single model of the world.
    It learns how the world changes across contexts.

---

## The problem

Through v11.6, the system carries three state dimensions:

    risk                  = 0.0 – 1.0
    reversibility         = 0.0 – 1.0
    epistemic_uncertainty = 0.0 – 1.0

These are treated as globally comparable. `risk = 0.5` means the same
thing regardless of where the system is operating.

It does not.

    risk = 0.5 in cybersecurity  →  moderate, probably manageable
    risk = 0.5 in clinical care  →  possibly unacceptable
    risk = 0.5 in industrial     →  depends on the process

Same number. Different meaning. Different admissibility. Different action.

The current architecture cannot distinguish these. It has one model of
what actions do, one set of admissibility thresholds, one evaluation function.

This is the scalar space problem Reuben identified:
values are treated as globally comparable rather than context-dependent.

---

## The shift

From:

    WorldState(risk, reversibility, epistemic_uncertainty)
    → single global model of effects and admissibility

To:

    Context(domain, risk_scale, reversibility_meaning, action_semantics)
    + WorldState
    → context-scoped model of effects and admissibility

The system does not learn one model. It learns per context.
Admissibility is not globally fixed. It is context-qualified.

---

## Context structure

```python
@dataclass
class Context:
    domain:               str         # "security" | "clinical" | "industrial" | ...
    risk_tolerance:       float       # 0.0 (zero tolerance) – 1.0 (high tolerance)
    reversibility_weight: float       # how much reversibility matters in this domain
    irreversible_eu_limit: float      # epistemic gate for irreversible actions
    operative_eu_limit:   float       # epistemic gate for operative actions
    action_semantics:     dict        # domain-specific meaning of each action
    description:          str
```

## Three domain examples

**Security**
```python
Context(
    domain="security",
    risk_tolerance=0.4,         # moderate — act before confirmed breach
    reversibility_weight=0.6,   # reversibility matters but less than in clinical
    irreversible_eu_limit=0.4,  # tighter — irreversible actions need more certainty
    operative_eu_limit=0.6,
    action_semantics={
        "collect_telemetry": "gather network/process evidence",
        "isolate_process":   "terminate and quarantine — hard to undo in production",
    }
)
```

**Clinical**
```python
Context(
    domain="clinical",
    risk_tolerance=0.2,         # low — patient safety is primary
    reversibility_weight=0.9,   # reversibility is near-absolute requirement
    irreversible_eu_limit=0.2,  # very tight — irreversible clinical acts need high certainty
    operative_eu_limit=0.5,
    action_semantics={
        "collect_telemetry": "gather diagnostic data — always safe",
        "isolate_process":   "intervention — may not be undoable",
    }
)
```

**Industrial**
```python
Context(
    domain="industrial",
    risk_tolerance=0.5,         # moderate — process disruption is costly but recoverable
    reversibility_weight=0.7,
    irreversible_eu_limit=0.45,
    operative_eu_limit=0.65,
    action_semantics={
        "collect_telemetry": "sensor reading / diagnostic cycle",
        "isolate_process":   "shutdown — significant operational cost",
    }
)
```

## What changes in admissibility

`is_admissible()` currently uses fixed thresholds:

```python
IRREVERSIBLE_EU_LIMIT = 0.5
OPERATIVE_EU_LIMIT    = 0.7
```

In v12.0, these come from the active context:

```python
def is_admissible(action, state, context):
    if atype == "irreversible":
        if state.epistemic_uncertainty > context.irreversible_eu_limit:
            return inadmissible(...)
```

The same epistemic uncertainty that is acceptable in security
is not acceptable in clinical. Context makes this explicit.

---

## What changes in evaluation

`evaluate_projected()` currently weights risk and reversibility equally
across all situations:

```python
risk_gain  = (current.risk - projected.risk) * 1.0
rev_gain   = (projected.rev - current.rev)   * 0.5
```

In v12.0, weights come from context:

```python
risk_gain = (current.risk - projected.risk)      * (1.0 / context.risk_tolerance)
rev_gain  = (projected.rev - current.rev)        * context.reversibility_weight
```

A low-risk-tolerance context (clinical) makes risk reduction more valuable.
A high-reversibility-weight context makes reversibility losses more costly.

---

## What changes in learning

In v11.5, the system maintains one `LearnedModel` with one set of learned adjustments.

In v12.0, learned adjustments are context-scoped:

```python
LearnedModel.learned_effects[context.domain][action][dimension]
```

What `collect_telemetry` does in security may differ from what it does in clinical.
The system learns these separately.

Cross-context generalisation is not automatic — it is a governed decision (v12.x).

---

## What does NOT change

    memi_decision()         — governance not affected
    VETO_1 / VETO_2         — constitutional, context-independent
    uncertainty_travels     — invariant
    LearningBuffer logic    — unchanged
    foresight structure     — unchanged

Constitutional vetos are not contextual. VETO_2 fires regardless of domain.
The epistemic gating principle is not contextual.
Context qualifies admissibility thresholds and evaluation weights.
It does not override governance.

---

## The key boundary

```
Context qualifies:    admissibility thresholds
                      evaluation weights
                      action semantics
                      learned effect scope

Context cannot touch: VETO conditions
                      governance authority thresholds
                      uncertainty_travels flag
                      base learning bounds
```

---

## Architectural invariant (updated)

    Foresight selects the proposal.
    Foresight only evaluates admissible proposals.
    Admissibility is qualified by context.
    Effect model explains why.
    Epistemic state qualifies confidence.
    Governance decides whether it is permitted.
    Decision payload carries uncertainty beyond the system.
    Learning proposals are governed decisions — not automatic updates.
    The system learns per context — not a single global model.

## Version plan

    v12.0  Context dataclass + context-qualified is_admissible()
           + context-weighted evaluate_projected()

    v12.1  Context-scoped LearnedModel
           (learned adjustments stored per domain)

    v12.2  Context switching
           (system detects or is told when context changes,
            switches learned model accordingly)

    v12.3  Cross-context generalisation as governed decision
           (proposal to transfer learning from one domain to another,
            subject to governance approval)

---

## The deeper shift

```
v11.x:
The system knows when it does not know enough to act.

v12.x:
The system knows that "enough" means different things in different contexts.
```
