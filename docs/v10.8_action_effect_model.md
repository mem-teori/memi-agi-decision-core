# M.E.M.I. v10.8 — Action Effect Model
## Architecture Note · April 2026

---

## The problem

In v10.7, actions are implicit:

```python
if action == "collect_telemetry":
    risk          -= 0.12
    reversibility += 0.12
```

The system selects `collect_telemetry` because it scores well.
It does not know *why* it scores well.
It does not know that the action is *epistemic* — that it works by
reducing uncertainty, not by directly reducing threat.

This means the system cannot:
- ask "what do I need right now?" and find the right action class
- distinguish between actions that address the same symptom differently
- reason about whether an action is appropriate for the current state type
- generalise to new actions without rewriting simulate()

---

## The shift

From:

```
action label → score
```

To:

```
action label → effect signature → score + semantic reasoning
```

An action is grounded when the system understands
what dimension of the world it changes.

---

## Action dimensions

The world state in v10.x has two dimensions:

| Dimension     | Meaning                                      |
|---------------|----------------------------------------------|
| `risk`        | Probability / severity of adverse outcome    |
| `reversibility` | Degree to which future actions remain open |

Actions affect these dimensions in different ways.
This is the grounding structure.

---

## Action type taxonomy

Three types, based on primary effect:

**Epistemic** — works by improving information quality
- Reduces uncertainty (proxied as risk via coverage)
- Increases reversibility (better data → more options)
- Example: `collect_telemetry`, `monitor`
- Key property: safe under high uncertainty — the action *reduces* the problem

**Operative** — works by directly changing the system state
- Reduces risk directly
- May reduce reversibility as a side effect
- Example: `reduce_load`, `isolate_process`
- Key property: requires sufficient authority — irreversible effects demand certainty

**Irreversible** — operative actions with permanent or hard-to-undo effects
- Highest risk reduction
- Highest reversibility cost
- Example: `isolate_process` (process kill, network isolation)
- Key property: MEMI veto conditions target this class specifically

## Effect signatures

```python
ACTION_MODEL = {
    "collect_telemetry": {
        "type":           "epistemic",
        "primary_effect": "reduce_uncertainty",
        "effects": {
            "risk":          -0.12,
            "reversibility": +0.12,
        },
        "conditions": {
            "effective_when": "uncertainty is high (risk > 0.4)",
            "diminishing_returns_when": "reversibility >= 0.9",
        },
    },
    "monitor": {
        "type":           "epistemic",
        "primary_effect": "reduce_uncertainty",
        "effects": {
            "risk":          -0.04,
            "reversibility": +0.04,
        },
        "conditions": {
            "effective_when": "always — minimal footprint",
        },
    },
    "reduce_load": {
        "type":           "operative",
        "primary_effect": "reduce_risk",
        "effects": {
            "risk":          -0.08,
            "reversibility":  0.00,
        },
        "conditions": {
            "effective_when": "risk > 0.3, reversibility preserved",
        },
    },
    "isolate_process": {
        "type":           "irreversible",
        "primary_effect": "reduce_risk",
        "effects": {
            "risk":          -0.15,
            "reversibility": -0.10,
        },
        "conditions": {
            "effective_when": "risk > 0.6 and reversibility > 0.4",
            "avoid_when":     "risk is already low — reversibility cost not justified",
        },
    },
}
```

## How foresight changes

In v10.7, `select_action_2step` scores actions numerically.

In v10.8, foresight can additionally:

1. **Filter by type** — under high uncertainty, prefer epistemic actions
2. **Check conditions** — avoid `isolate_process` when `reversibility < 0.4`
3. **Explain selection** — "selected `collect_telemetry` because type=epistemic and uncertainty is high"

The score still drives the final selection.
The effect model adds a semantic layer on top of the score.

---

## What changes in the code

**New:** `ACTION_MODEL` dict — effect signatures per action

**New:** `simulate_from_model(state, action)` — uses ACTION_MODEL instead of hardcoded if/elif

**New:** `semantic_filter(candidates, state)` — deprioritises actions that are inappropriate for current state

**New:** `explain_selection(action, state, score)` — produces a readable reason string

**Unchanged:** `memi_decision()` — governance is not affected
**Unchanged:** `select_action_2step()` — foresight structure is not affected
**Unchanged:** all trajectory, evaluation, and loop logic

---

## The architectural invariant

The effect model is a *proposal filter and explainer*.
It does not change what MEMI permits.
It changes what the system understands about its own proposals.

```
Foresight selects the proposal.
Effect model explains why.
Governance decides whether it is permitted.
```
